# AI 에이전트와 함께 MVP 만들기 — dooray-cli 사례

## 이 문서는

Claude Code(AI 코딩 에이전트)를 활용해 CLI 도구의 MVP를 빠르게 구축한 실제 사례를 정리한 가이드입니다.

"대화로 컨텍스트를 쌓고 → 문서로 정제하고 → 에이전트가 자율 실행"하는 패턴을 단계별로 설명합니다.

### 대상 독자

- Claude Code를 사용하는 개발자
- AI 에이전트로 프로젝트를 자동화하고 싶은 분
- "프롬프트 한 줄로 앱을 만든다"가 아닌, 체계적 접근법이 궁금한 분

### 핵심 메시지

> 에이전트가 잘 동작하는 이유는 코드를 잘 짜서가 아니라, **실행 전에 문서가 충분히 정제되어 있기 때문**이다.

---

## 전체 흐름

```
Phase 0: 컨텍스트 빌딩     — 대화로 기술적 결정을 쌓는다
Phase 1: 문서 완성          — 놓친 점을 반복 검토하여 문서를 정제한다
Phase 2: 하네스 구축        — 문서를 기반으로 실행 계획을 phase로 분리한다
Phase 3: 자율 실행          — 에이전트가 코드를 구현하고 검증한다
```

각 Phase는 **이전 Phase의 산출물에 의존**한다. Phase 0~1이 부실하면 Phase 3에서 에이전트가 방향을 잃는다.

---

## Phase 0: 컨텍스트 빌딩

### 목적

에이전트가 "무엇을 왜 어떻게" 만들어야 하는지 이해할 수 있는 양질의 컨텍스트를 대화로 만든다.

### 핵심 프롬프트

#### 1. 프로젝트 정의

```
"Dooray REST API를 CLI로 래핑해서 AI 에이전트와 터미널 사용자가 
Dooray를 조작할 수 있게 하고 싶어. MCP 도구 대신 CLI가 나은 이유는..."
```

**왜 주효했는가**: 단순히 "CLI 만들어줘"가 아니라 **문제(MCP의 한계)와 해결 방향(CLI)을 함께 제시**했다. 에이전트가 설계 결정을 내릴 때 이 맥락을 참조할 수 있다.

#### 2. 기술 스택 논의

```
"TypeScript + Commander.js 기반으로 가려고 해. HTTP 클라이언트는 ky가 좋을까? 
빌드는 tsup으로 단일 번들? 의견 줘봐"
```

**왜 주효했는가**: 후보를 먼저 제시하고 에이전트의 의견을 구했다. 에이전트가 각 선택지의 장단점을 분석하고, 그 결과를 ADR(Architecture Decision Record)로 기록하는 흐름이 자연스럽게 만들어졌다.

#### 3. 범위 확정

```
"MVP 범위를 정하자. v1에는 project, post, comment, wiki CRUD만 넣고
파일 첨부, 마일스톤 관리는 제외하려고 해"
```

**왜 주효했는가**: 범위를 명시적으로 정의함으로써 에이전트가 "이것도 넣을까요?"라는 불필요한 제안을 하지 않게 된다. **제외 항목을 명시하는 것이 포함 항목만큼 중요하다.**

### 산출물

이 Phase에서 만들어진 문서:

| 문서 | 내용 | 역할 |
|------|------|------|
| `docs/prd.md` | 한 줄 정의, 문제, 타겟, 범위 | 에이전트의 "왜" |
| `docs/adr.md` | 기술 결정 + 이유 | 에이전트의 "어떻게" |
| `docs/code-architecture.md` | 디렉토리 구조, 모듈 의존 | 에이전트의 "어디에" |
| `docs/data-schema.md` | config, 캐시, resolver 스키마 | 에이전트의 "무엇을" |
| `docs/flow.md` | 사용자 흐름 | 에이전트의 "어떤 순서로" |

## Phase 1: 문서 완성 — 반복 검토 루프

### 목적

Phase 0의 문서에서 빠진 부분을 찾아 채운다. 이 루프가 문서의 품질을 결정하고, 문서의 품질이 자율 실행의 성패를 결정한다.

### 핵심 프롬프트

#### 검토 요청

```
"docs 폴더를 전체적으로 모두 확인한 거 맞아? 
docs 폴더를 순차적으로 먼저 읽고 업데이트 진행한 거야?"
```

**왜 주효했는가**: 에이전트는 효율을 위해 관련 파일만 읽는 경향이 있다. **"전체를 확인했느냐"고 명시적으로 물어야** 놓친 파일(이 경우 `prd.md`, `flow.md`)을 발견할 수 있다.

#### 기술적 평가 요청

```
"api/types.ts 파일이 점점 커지고 있는데 types 폴더를 만들어서 분리하는 건 어때?
구글 시니어 엔지니어로서 기술적 평가를 부탁할게"
```

**왜 주효했는가**: 역할을 부여하면 에이전트가 찬성/반대 양면을 분석하고, 임계점 기준(예: 800줄)과 재검토 시점까지 제안한다. 결과를 ADR로 기록하면 향후 같은 논의를 반복하지 않는다.

#### 결정 → 기록 루프

```
"오케이 그럼 아직 분리하지 않아도 되겠네. 이 논의까지 포함해서 docs 폴더를 업데이트하자"
```

**패턴**: 논의 → 결정 → 문서 반영. 이 루프를 모든 기술적 결정에 적용한다.

### 검토 루프 다이어그램

```
┌─────────────────┐
│ 문서 전체 읽기   │
└────────┬────────┘
         ▼
┌─────────────────┐     ┌──────────────┐
│ 빠진 부분 있나?  │──▶  │ 논의 + 결정   │
└────────┬────────┘     └──────┬───────┘
         │ 없음                │
         ▼                     ▼
┌─────────────────┐     ┌──────────────┐
│ Phase 1 완료     │     │ 문서에 반영   │──┐
└─────────────────┘     └──────────────┘  │
         ▲                                 │
         └─────────────────────────────────┘
```

## Phase 2: 하네스 구축

### "하네스"란?

에이전트가 자율적으로 실행할 수 있도록 **계획, 실행 순서, 완료 조건을 정의한 구조**를 말한다. 마치 테스트 하네스가 테스트 실행 환경을 제공하듯, 에이전트 하네스는 에이전트의 실행 환경을 제공한다.

### 하네스의 핵심 구성 요소

```
1. Phase 정의      — 무엇을 만들 것인가 (범위)
2. 실행 순서        — 어떤 순서로 만들 것인가 (의존성)
3. 완료 조건        — 언제 끝난 것인가 (검증)
4. 컨텍스트 참조    — 어디를 보고 만들 것인가 (문서)
```

### dooray-cli의 Phase 분리

```yaml
phases:
  - name: "v0.1.0 — Core CRUD"
    scope:
      - config/doctor/cache 커맨드
      - project 조회 (목록, 멤버, 워크플로우)
      - post CRUD (목록, 검색, 조회, 생성, 수정, 완료, 상태변경)
      - comment CRUD (목록, 추가, 수정, 삭제)
      - wiki (목록, 페이지 조회, 생성, 수정)
    context:
      - docs/prd.md (MVP 범위)
      - docs/code-architecture.md (디렉토리 구조)
      - docs/data-schema.md (캐시, config 스키마)
      - docs/adr.md (기술 결정)
    completion:
      - pnpm run build 성공
      - dooray --help 출력 확인
      - dooray doctor 실행 성공 (API 연결 검증)
      - dooray post list <project> 실행 성공
    
  - name: "v0.2.0 — Mail"
    scope:
      - IMAP 메일 조회 (목록, 상세, 검색, 안읽은 메일)
      - SMTP 메일 발송 (발송, 답장)
      - config에 IMAP/SMTP 설정 추가
    context:
      - docs/adr.md (ADR-012 IMAP, ADR-013 SMTP)
      - docs/flow.md (메일 흐름)
    completion:
      - pnpm run build 성공
      - dooray mail list 실행 성공
      - dooray mail send 실행 성공

  - name: "v0.3.0 — File Attachment"
    scope:
      - 첨부파일 목록/다운로드/업로드/삭제
      - 307 리다이렉트 수동 처리
    context:
      - docs/adr.md (ADR-015 파일 리다이렉트)
      - Dooray 파일 API 문서 (사용자 제공)
    completion:
      - pnpm run build 성공
      - dooray post file list <project> <number> 실행 성공
      - dooray post file download 실행 성공
      - dooray post file upload 실행 성공
```

### Phase 분리의 핵심 원칙

1. **각 Phase는 독립 배포 가능해야 한다** — v0.1.0 없이 v0.2.0은 불가하지만, v0.2.0 없이 v0.1.0은 완전히 동작한다
2. **완료 조건은 자동 검증 가능해야 한다** — "코드가 깔끔한가"가 아니라 "빌드 성공 + CLI 실행 성공"처럼 기계적으로 판단 가능한 조건
3. **컨텍스트 참조를 명시해야 한다** — 에이전트가 어떤 문서를 읽고 실행해야 하는지 알려줘야 한다

### SKILL.md로 하네스 구성

Claude Code에서는 스킬 파일로 하네스를 구성할 수 있다.

```
.claude/skills/
  plan-and-build/
    SKILL.md        # 하네스 정의 (Phase 목록, 실행 순서, 완료 조건)
```

스킬 파일 예시 (`.claude/skills/plan-and-build/SKILL.md`):

```markdown
---
name: plan-and-build
description: docs/ 폴더의 문서를 기반으로 Phase별 순차 구현을 실행
---

# plan-and-build

## 실행 절차

1. docs/ 폴더의 모든 문서를 읽어 프로젝트 컨텍스트를 파악한다
2. 현재 구현 상태를 파악한다 (git log, 디렉토리 구조)
3. 다음 미완성 Phase를 식별한다
4. 해당 Phase의 scope에 따라 코드를 구현한다
5. 완료 조건을 검증한다 (빌드, CLI 실행)
6. 검증 통과 시 다음 Phase로 진행한다
7. 모든 Phase 완료 시 종료한다

## 규칙

- 각 Phase 시작 전 반드시 context에 명시된 문서를 읽는다
- 완료 조건을 모두 통과해야 다음 Phase로 넘어간다
- 완료 조건 실패 시 자동으로 디버깅을 시도한다
- 3회 연속 실패 시 사용자에게 개입을 요청한다
```

## Phase 3: 자율 실행

### 에이전트가 자율적으로 동작할 수 있는 이유

에이전트가 코드를 잘 짜서가 아니다. **에이전트가 참조할 수 있는 문서가 충분히 정제되어 있기 때문이다.**

자율 실행이 가능했던 핵심 요인:

#### 1. 문서가 "왜"를 포함한다

```
# BAD — 무엇만 기록
"HTTP 클라이언트로 ky를 사용한다"

# GOOD — 왜를 함께 기록 (ADR)
"HTTP 클라이언트로 ky를 사용한다
 이유: fetch 기반으로 경량이고 axios 대비 번들 크기 작음
 재검토 시점: interceptor가 필요해지면 axios 검토"
```

"왜"가 있으면 에이전트가 엣지 케이스에서 스스로 판단할 수 있다. 예를 들어 파일 업로드에서 `ky`의 redirect 처리가 안 될 때, ADR에 "ky는 fetch 기반"이라는 맥락이 있으므로 `fetch`를 직접 사용하는 우회를 스스로 시도할 수 있다.

#### 2. 완료 조건이 기계적이다

```
# BAD — 주관적 완료 조건
"파일 다운로드 기능이 잘 동작하면 완료"

# GOOD — 기계적 완료 조건
"pnpm run build 성공 + dooray post file download tc-ocr 454 <file-id> 실행 시 파일 저장됨"
```

에이전트는 "잘 동작하는지"를 판단할 수 없지만, "빌드 성공 여부"와 "명령 실행 결과"는 판단할 수 있다.

#### 3. 실패 시 복구 경로가 있다

dooray-cli에서 실제 발생한 사례:

```
문제: 파일 다운로드 API가 404를 반환
원인: /files/{id}/downloads 경로가 아니라 /files/{id}?media=raw 였음
복구: 에이전트가 다양한 URL 패턴을 시도 → 정상 경로 발견 → ADR에 기록
```

이것이 가능했던 이유는 **API 문서가 컨텍스트에 포함되어 있었기 때문**이다. 에이전트가 "이 API는 `?media=raw`와 `?media=meta`를 사용한다"는 정보를 문서에서 찾아 스스로 해결했다.

### 실행 프롬프트

하네스가 준비되면 실행은 간단하다:

```
/plan-and-build
```

또는 직접 지시:

```
"docs/ 폴더를 모두 읽고, Phase 분리에 따라 v0.1.0부터 순차적으로 구현해줘.
각 Phase마다 빌드 검증하고, 실패하면 디버깅해서 통과시켜."
```

## 실제 사례: dooray-cli 타임라인

### v0.1.0 — Core CRUD (첫 대화)

```
[대화 1] 프로젝트 정의 + 기술 스택 논의
         → prd.md, adr.md, code-architecture.md 생성

[대화 2] 범위 확정 + 데이터 스키마 논의
         → data-schema.md, flow.md 생성

[대화 3] /plan-and-build 실행
         → 에이전트가 src/ 전체 구현
         → config → cache → resolver → api/client → commands 순서로 구현
         → 빌드 성공 + CLI 동작 확인
         → npm publish v0.1.0
```

### v0.2.0 — Mail (두 번째 대화)

```
[대화 4] "메일 연동도 하고 싶은데"
         → IMAP/SMTP 기술 논의
         → Dooray IMAP 서버 한계 발견 (SINCE 미지원, SORT 미지원)
         → ADR-012, ADR-013 기록

[대화 5] 에이전트가 구현 + 디버깅
         → imapflow UID 파라미터 이슈 자체 해결
         → tsup external 설정 이슈 자체 해결
         → npm publish v0.2.0
```

### v0.3.0 — File Attachment (세 번째 대화)

```
[대화 6] "파일 첨부 기능을 추가하고 싶어" + API 문서 제공
         → 307 리다이렉트 패턴 논의
         → ADR-015 기록

[대화 7] 에이전트가 구현
         → 다운로드 404 오류 → ?media=raw 경로 자체 발견
         → 업로드 ky redirect 이슈 → fetch 직접 사용으로 우회
         → 실제 API 테스트 전부 통과
         → npm publish v0.3.0
```

### 타임라인에서 보이는 패턴

1. **대화가 진행될수록 사람의 개입이 줄어든다** — v0.1.0은 논의 비중이 컸지만, v0.3.0은 API 문서만 전달하면 에이전트가 자체 해결
2. **문서가 누적되면서 에이전트의 판단력이 올라간다** — ADR, 스키마, 아키텍처 문서가 쌓일수록 에이전트가 맥락을 더 잘 이해
3. **디버깅도 자율적으로 수행** — 에러 → 원인 분석 → URL 패턴 시도 → 해결 → ADR 기록까지 자동

## 핵심 원칙 요약

### 1. 문서가 코드보다 먼저

코드를 작성하기 전에 PRD, ADR, 아키텍처 문서를 먼저 만든다. 에이전트가 참조할 수 있는 컨텍스트가 없으면 자율 실행은 불가능하다.

### 2. "왜"를 기록한다

모든 기술적 결정은 ADR로 기록한다. 결정뿐 아니라 **이유와 재검토 시점**을 함께 기록해야 에이전트가 엣지 케이스에서 판단할 수 있다.

### 3. Phase는 독립 배포 단위로

각 Phase가 끝나면 독립적으로 동작하는 제품이어야 한다. 이래야 중간에 실패해도 이전 Phase의 결과물은 보존된다.

### 4. 완료 조건은 기계적으로

"잘 동작하면 완료"가 아니라 "빌드 성공 + 이 명령 실행 시 이 결과"처럼 에이전트가 자동으로 판단할 수 있는 조건을 정의한다.

### 5. 놓친 점은 반드시 물어본다

에이전트는 효율을 추구하느라 일부를 건너뛸 수 있다. "전체 다 확인했어?"라고 명시적으로 물어야 빠진 부분을 발견할 수 있다.

### 6. 논의 → 결정 → 기록 루프

기술적 논의가 끝나면 반드시 결정을 내리고, 결정을 문서에 반영한다. 문서에 없는 결정은 다음 대화에서 잊혀진다.

## 시작하기

당신의 프로젝트에 이 방법론을 적용하려면:

### Step 1: docs/ 폴더 만들기

```bash
mkdir docs
```

### Step 2: Claude Code와 대화하면서 문서 채우기

```
"나는 [문제]를 해결하기 위해 [도구]를 만들려고 해.
타겟 사용자는 [누구]이고, 기술 스택은 [이것]을 고려 중이야.
PRD를 먼저 작성해보자."
```

### Step 3: 문서 검토 루프

```
"docs 폴더 전체를 읽고, 빠진 부분이 있는지 확인해줘"
```

→ 빠진 부분이 없을 때까지 반복

### Step 4: Phase 분리

```
"이제 구현을 Phase로 나누자. 
각 Phase는 독립 배포 가능하고, 완료 조건은 기계적으로 정의해줘"
```

### Step 5: 실행

```
"docs/ 폴더를 모두 읽고, Phase 순서대로 구현해줘.
각 Phase마다 빌드 검증하고, 완료 조건을 확인해."
```

이것이 전부다. 문서의 품질이 곧 결과물의 품질이다.
