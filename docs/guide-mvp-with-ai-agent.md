# AI 에이전트와 함께 MVP 만들기 — dooray-cli 사례

## 결론부터

**양질의 컨텍스트 → 문서화 → 하네스(자율 실행 구조)**

이 세 단계를 거치면 AI 에이전트가 사람의 개입 없이 MVP를 구현할 수 있다. 프롬프트 한 줄로 앱을 만드는 것이 아니다. **에이전트가 자율적으로 동작할 수 있는 환경을 체계적으로 구축**하는 것이다.

이 문서는 Claude Code로 dooray-cli(Dooray REST API CLI 도구)의 MVP를 구축한 실제 사례를 바탕으로, 이 방법론을 단계별로 설명한다.

> 에이전트가 잘 동작하는 이유는 코드를 잘 짜서가 아니라, **실행 전에 문서가 충분히 정제되어 있기 때문**이다.

---

## 하네스(Harness)란?

테스트 하네스가 테스트의 실행 환경을 제공하듯, **에이전트 하네스는 에이전트의 자율 실행 환경을 제공하는 구조**다.

```
하네스 = 실행 계획 + 완료 조건 + 컨텍스트 참조
```

구체적으로는:

- **실행 계획**: 무엇을 어떤 순서로 만들 것인가 (Phase 분리)
- **완료 조건**: 각 Phase가 언제 끝난 것인가 (빌드 성공, CLI 실행 등 기계적 판단 가능한 조건)
- **컨텍스트 참조**: 에이전트가 어떤 문서를 읽고 실행해야 하는가

하네스가 없으면 에이전트는 "다음에 뭘 해야 하지?"를 매번 사람에게 물어야 한다. 하네스가 있으면 에이전트가 스스로 판단하고 진행한다.

---

## 전체 흐름

```
1. 양질의 컨텍스트   — 대화로 기술적 결정을 깊이 있게 쌓는다
2. 문서화             — 대화의 결과를 5개 문서로 정제한다
3. 하네스 구축        — 문서를 기반으로 Phase를 분리하고 완료 조건을 정의한다
4. 자율 실행          — 에이전트가 하네스에 따라 코드를 구현하고 검증한다
```

각 단계는 **이전 단계의 산출물에 의존**한다. 1~2가 부실하면 4에서 에이전트가 방향을 잃는다.

### 대상 독자

- Claude Code를 사용하는 개발자
- AI 에이전트로 프로젝트를 자동화하고 싶은 분
- "프롬프트 한 줄로 앱을 만든다"가 아닌, 체계적 접근법이 궁금한 분

## 1. 양질의 컨텍스트 — 대화로 쌓기

### 목적

에이전트가 "무엇을 왜 어떻게" 만들어야 하는지 이해할 수 있는 양질의 컨텍스트를 대화로 만든다. 이 Phase가 전체 프로세스에서 **가장 중요**하다. 여기서 쌓은 컨텍스트의 품질이 자율 실행의 성패를 결정한다.

### 9단계 대화 워크플로우

컨텍스트 빌딩은 하나의 대화 세션에서 9단계를 순차적으로 진행한다. 각 단계마다 **에이전트에게 역할을 부여**하고, **"더 논의할 부분 없어?"를 반복**하며, 모호한 점이 완전히 해소될 때까지 다음 단계로 넘어가지 않는다.

#### Step 1: 구현 가능성 검증

```
"지금부터 위 요구사항을 구현하기 위한 설계를 시작할 것이다. 
첫 단계는 구현 가능성 검증이다. 이 서비스가 실제로 구현 가능할지 기술적으로 검증해야 한다.

너는 나의 공동창업자고, 우리의 목표는 빠르게 스타트업을 만들어 
다음 Y Combinator batch에 합격하는 거다. 그걸 위해서 우리는 빠르게 MVP를 출시하고, 
시장의 반응 보면서 여러 가설을 테스트해야 한다. 속도가 생명이다. 
그러면서도 안정적인 서비스를 만들어야 한다. 이 둘간의 트레이드오프를 명심해라.

그럼 지금부터 논의를 시작하자. 네가 주도적으로 구현 가능성을 점검하고, 
조금이라도 모호하거나 구체화해야 할 것이 있다면 나와 논의하자."
```

**왜 주효했는가**:
- **역할 부여** ("공동창업자") — 에이전트가 단순 실행자가 아닌 의사결정 참여자로 행동하게 만든다
- **가치관 명시** ("속도 vs 안정성 트레이드오프") — 이후 모든 결정에서 이 기준이 적용된다
- **주도성 부여** ("네가 주도적으로") — 에이전트가 수동적으로 대답만 하는 게 아니라 능동적으로 질문하고 문제를 제기하게 된다

#### Step 2: 기술 스택 선정

```
"TypeScript + Commander.js 기반으로 가려고 해. HTTP 클라이언트는 ky가 좋을까? 
빌드는 tsup으로 단일 번들? 의견 줘봐"
```

그리고 에이전트가 의견을 제시하면:

```
"더 논의할 부분은 없어? 진짜 넘어가도 돼?"
```

**왜 주효했는가**:
- 후보를 먼저 제시하면 에이전트가 각 선택지의 장단점을 분석한다
- **"진짜 넘어가도 돼?"** — 이 한 마디가 에이전트로 하여금 놓친 점을 한 번 더 점검하게 만든다. 이 패턴은 모든 단계에서 반복한다
- 결과를 ADR(Architecture Decision Record)로 기록하는 흐름이 자연스럽게 형성된다

#### Step 3: 사용 흐름 점검

```
"이제 사용자 흐름을 논의하고 싶어. 구현 가능성 때처럼 조금이라도 모호하거나 
구체화하고 싶은 게 있으면 내게 말해주면 돼.

참고로 너는 이제부터 구글 시니어 UX리서처로서 응답하면 돼. 
이번 논의에서 네가 지향해야 할 것은, 이 앱의 주요 사용 흐름이 너의 머릿속에 
생생히 재현돼서 그대로 앱으로 바로 옮길 수 있을 정도로 명확해지는 거야. 
이 수준에 도달하지 못했다면 전부 잡아내야 돼."
```

**왜 주효했는가**:
- **역할 전환** ("구글 시니어 UX리서처") — 기술 논의에서 사용자 관점으로 전환
- **완료 기준 명시** ("머릿속에 생생히 재현") — 에이전트가 추상적으로 넘어가지 않고 구체적인 흐름을 점검하게 된다
- CLI 프로그램의 경우 화면 설계 대신 **명령어 흐름**과 **출력 포맷**을 이 단계에서 점검했다

> **참고**: dooray-cli는 CLI 프로그램이라 별도의 화면 설계 단계는 생략했다. 웹/앱 프로젝트라면 이 단계에서 화면별 정보와 기능을 점검한다:
> ```
> "이 서비스에 필요한 화면들에 대해서, 그 화면에 어떤 정보가 표시되고 
> 어떤 기능이 있을 것인지를 점검하면 돼. 
> 이 단계에서도 뭔가 모호한 부분이 있다면 나한테 말해야 돼"
> ```
> 그리고 마지막에 반드시:
> ```
> "진짜 더 얘기할 부분 없어? 마지막으로 한번 더 체크 좀"
> ```

#### Step 4: API 설계

```
"이제 API 설계를 하자. Dooray REST API의 엔드포인트를 기반으로
CLI가 호출해야 할 API 목록과 요청/응답 구조를 정리하자"
```

**왜 주효했는가**: 외부 API를 래핑하는 프로젝트에서는 API 구조가 전체 설계의 기반이 된다. 이 단계에서 요청/응답 타입을 명확히 정의하면, 이후 `api/types.ts` 구현이 자동으로 이루어진다.

#### Step 5: 데이터 설계

```
"데이터 스키마 설계랑 코드 아키텍처 설계를 같이 해볼 거야. 
이제 다시 마이크로소프트 시니어 개발자 출신의 CTO로 돌아오면 돼. 

지금까지 논의된 앱을 구현하기 위한 전반적 설계를 해보고, 
조금이라도 좀 모호해서 나랑 논의해서 결정해야 할 것이 있다면 전부 알려줘. 
우리 가치관 기억하지? 최대한 간결하고 빠르게 출시하는 거야. 이거 명심해! 
그럼 이제 시작하자"
```

**왜 주효했는가**:
- **역할 재전환** ("MS 시니어 개발자 출신 CTO") — 사용자 관점에서 다시 기술 관점으로 전환
- **가치관 재확인** ("간결하고 빠르게") — 에이전트가 과도한 설계를 하지 않도록 제어
- config, 캐시 구조, resolver 패턴 등 런타임 데이터 흐름이 이 단계에서 결정되었다

#### Step 6: 코드 아키텍처 설계

데이터 설계와 이어서 디렉토리 구조, 모듈 의존 관계, 에러 처리 전략을 논의한다.

```
commands/* → resolvers/* → cache/store + api/client
commands/* → formatters/*
commands/* → utils/errors
```

이 의존 관계가 명확해야 에이전트가 구현 순서를 스스로 결정할 수 있다.

#### Step 7: 기술적 결정사항 점검

```
"지금까지 논의한 기술적 결정사항들을 정리하자. 빠진 것 없는지 전부 확인해줘"
```

**왜 주효했는가**: 여기까지 오면 수많은 결정이 대화 중에 흩어져 있다. 이 단계에서 명시적으로 모아야 ADR 문서로 정제할 수 있다.

#### Step 8: 최종 검토

```
"더 논의할 거 있어? 진짜 없어?"
```

모든 단계를 마친 후 마지막으로 한 번 더 확인한다. 이 시점에서 에이전트가 "사실 이 부분이 좀 걱정되는데..."라고 꺼내는 것이 있다면, 그것이 바로 문서화되어야 할 핵심이다.

#### Step 9: 최종 문서 생성

```
"좋아. 고생했어. 이제 우리가 지금까지 논의한 내용을 바탕으로 문서 생성하자.

다음 문서들이 필요하다:
prd, flow (사용자 흐름), data-schema, code-architecture, adr (기술적 결정사항)

이 문서를 만들 때는, 다음 주의사항을 반드시 숙지해서 적절한 내용으로 구성해라. 
이 문서는 AI 에이전트를 위한 것이다. 그들의 컨텍스트를 낭비하지 않도록, 
같은 내용도 더 간결하게 쓸 수 있도록 해야 한다. 간결한 문서를 추구하되, 
의사결정 의도와 같은 맥락적 정보가 아예 제거되어서도 안 된다. 
이 문서를 보고 AI 에이전트가 우리의 기획적, 개발적 철학을 온전히 추론할 수 있어야 한다."
```

**왜 주효했는가**:
- **문서의 대상을 명시** ("AI 에이전트를 위한 것") — 사람을 위한 장황한 문서가 아닌, 에이전트가 효율적으로 소비할 수 있는 간결한 문서를 생성하게 만든다
- **간결함과 맥락 보존의 균형** — "간결하되 의사결정 의도는 보존"이라는 기준이 있어야 에이전트가 적절한 수준의 문서를 생산한다
- 이 대화에서 쌓인 모든 컨텍스트가 5개 문서로 정제된다

### 9단계의 핵심 패턴

위 9단계를 관통하는 3가지 패턴:

**1. 역할 부여로 관점을 전환한다**

| 단계 | 역할 | 관점 |
|------|------|------|
| 구현 가능성 | 공동창업자 | 비즈니스 + 기술 |
| 사용 흐름 | 구글 시니어 UX리서처 | 사용자 경험 |
| 데이터/아키텍처 | MS 시니어 개발자 CTO | 기술 설계 |
| 기술 평가 | 구글 시니어 엔지니어 | 코드 품질 |

역할을 부여하면 에이전트가 해당 관점에서 깊이 있게 분석하고, 그 관점에서만 보이는 문제를 제기한다.

**2. "더 논의할 거 없어?"를 반복한다**

모든 단계의 끝에서 이 질문을 던진다. 에이전트는 효율을 위해 "충분하다"고 빨리 넘어가려는 경향이 있다. 이 질문이 **품질 게이트** 역할을 한다.

**3. 논의 → 결정 → 기록을 순환한다**

```
대화에서 논의 → "이걸로 가자" 결정 → 문서에 반영 → 다음 주제
```

문서에 반영되지 않은 결정은 다음 대화에서 잊혀진다. 반드시 기록한다.

### 산출물

이 Phase에서 만들어진 문서:

| 문서 | 내용 | 역할 |
|------|------|------|
| `docs/prd.md` | 한 줄 정의, 문제, 타겟, 범위 | 에이전트의 "왜" |
| `docs/adr.md` | 기술 결정 + 이유 + 재검토 시점 | 에이전트의 "어떻게" |
| `docs/code-architecture.md` | 디렉토리 구조, 모듈 의존 | 에이전트의 "어디에" |
| `docs/data-schema.md` | config, 캐시, resolver 스키마 | 에이전트의 "무엇을" |
| `docs/flow.md` | 사용자 흐름 | 에이전트의 "어떤 순서로" |

## 2. 문서화 — 컨텍스트를 정제하기

### 목적

컨텍스트 빌딩 단계의 문서에서 빠진 부분을 찾아 채운다. 이 루프가 문서의 품질을 결정하고, 문서의 품질이 자율 실행의 성패를 결정한다.

### 핵심 프롬프트

#### 검토 요청

```
"docs 폴더를 전체적으로 모두 확인한 거 맞아? 
docs 폴더를 순차적으로 먼저 읽고 업데이트 진행한 거야?"
```

**왜 주효했는가**: 에이전트는 효율을 위해 관련 파일만 읽는 경향이 있다. **"전체를 확인했느냐"고 명시적으로 물어야** 놓친 파일(이 경우 `prd.md`, `flow.md`)을 발견할 수 있다.

#### 기술적 평가 요청

```
"api/types.ts 파일이 점점 커지고 있는데 types 폴더를 만들어서 분리하는 건 어때?
구글 시니어 엔지니어로서 기술적 평가를 부탁할게"
```

**왜 주효했는가**: 역할을 부여하면 에이전트가 찬성/반대 양면을 분석하고, 임계점 기준(예: 800줄)과 재검토 시점까지 제안한다. 결과를 ADR로 기록하면 향후 같은 논의를 반복하지 않는다.

#### 결정 → 기록 루프

```
"오케이 그럼 아직 분리하지 않아도 되겠네. 이 논의까지 포함해서 docs 폴더를 업데이트하자"
```

**패턴**: 논의 → 결정 → 문서 반영. 이 루프를 모든 기술적 결정에 적용한다.

### 검토 루프 다이어그램

## 3. 하네스 구축 — 자율 실행 환경 만들기

### 하네스의 핵심 구성 요소

```
1. Phase 정의      — 무엇을 만들 것인가 (범위)
2. 실행 순서        — 어떤 순서로 만들 것인가 (의존성)
3. 완료 조건        — 언제 끝난 것인가 (검증)
4. 컨텍스트 참조    — 어디를 보고 만들 것인가 (문서)
```

### dooray-cli의 Phase 분리

```yaml
phases:
  - name: "v0.1.0 — Core CRUD"
    scope:
      - config/doctor/cache 커맨드
      - project 조회 (목록, 멤버, 워크플로우)
      - post CRUD (목록, 검색, 조회, 생성, 수정, 완료, 상태변경)
      - comment CRUD (목록, 추가, 수정, 삭제)
      - wiki (목록, 페이지 조회, 생성, 수정)
    context:
      - docs/prd.md (MVP 범위)
      - docs/code-architecture.md (디렉토리 구조)
      - docs/data-schema.md (캐시, config 스키마)
      - docs/adr.md (기술 결정)
    completion:
      - pnpm run build 성공
      - dooray --help 출력 확인
      - dooray doctor 실행 성공 (API 연결 검증)
      - dooray post list <project> 실행 성공
    
  - name: "v0.2.0 — Mail"
    scope:
      - IMAP 메일 조회 (목록, 상세, 검색, 안읽은 메일)
      - SMTP 메일 발송 (발송, 답장)
      - config에 IMAP/SMTP 설정 추가
    context:
      - docs/adr.md (ADR-012 IMAP, ADR-013 SMTP)
      - docs/flow.md (메일 흐름)
    completion:
      - pnpm run build 성공
      - dooray mail list 실행 성공
      - dooray mail send 실행 성공

  - name: "v0.3.0 — File Attachment"
    scope:
      - 첨부파일 목록/다운로드/업로드/삭제
      - 307 리다이렉트 수동 처리
    context:
      - docs/adr.md (ADR-015 파일 리다이렉트)
      - Dooray 파일 API 문서 (사용자 제공)
    completion:
      - pnpm run build 성공
      - dooray post file list <project> <number> 실행 성공
      - dooray post file download 실행 성공
      - dooray post file upload 실행 성공
```

### Phase 분리의 핵심 원칙

1. **각 Phase는 독립 배포 가능해야 한다** — v0.1.0 없이 v0.2.0은 불가하지만, v0.2.0 없이 v0.1.0은 완전히 동작한다
2. **완료 조건은 자동 검증 가능해야 한다** — "코드가 깔끔한가"가 아니라 "빌드 성공 + CLI 실행 성공"처럼 기계적으로 판단 가능한 조건
3. **컨텍스트 참조를 명시해야 한다** — 에이전트가 어떤 문서를 읽고 실행해야 하는지 알려줘야 한다

### SKILL.md로 하네스 구성

Claude Code에서는 스킬 파일로 하네스를 구성할 수 있다.

```
.claude/skills/
  plan-and-build/
    SKILL.md        # 하네스 정의 (Phase 목록, 실행 순서, 완료 조건)
```

스킬 파일 예시 (`.claude/skills/plan-and-build/SKILL.md`):

```markdown
---
name: plan-and-build
description: docs/ 폴더의 문서를 기반으로 Phase별 순차 구현을 실행
---

# plan-and-build

## 실행 절차

1. docs/ 폴더의 모든 문서를 읽어 프로젝트 컨텍스트를 파악한다
2. 현재 구현 상태를 파악한다 (git log, 디렉토리 구조)
3. 다음 미완성 Phase를 식별한다
4. 해당 Phase의 scope에 따라 코드를 구현한다
5. 완료 조건을 검증한다 (빌드, CLI 실행)
6. 검증 통과 시 다음 Phase로 진행한다
7. 모든 Phase 완료 시 종료한다

## 규칙

- 각 Phase 시작 전 반드시 context에 명시된 문서를 읽는다
- 완료 조건을 모두 통과해야 다음 Phase로 넘어간다
- 완료 조건 실패 시 자동으로 디버깅을 시도한다
- 3회 연속 실패 시 사용자에게 개입을 요청한다
```

## 4. 자율 실행 — 에이전트가 코드를 구현한다

### 에이전트가 자율적으로 동작할 수 있는 이유

에이전트가 코드를 잘 짜서가 아니다. **에이전트가 참조할 수 있는 문서가 충분히 정제되어 있기 때문이다.**

자율 실행이 가능했던 핵심 요인:

#### 1. 문서가 "왜"를 포함한다

```
# BAD — 무엇만 기록
"HTTP 클라이언트로 ky를 사용한다"

# GOOD — 왜를 함께 기록 (ADR)
"HTTP 클라이언트로 ky를 사용한다
 이유: fetch 기반으로 경량이고 axios 대비 번들 크기 작음
 재검토 시점: interceptor가 필요해지면 axios 검토"
```

"왜"가 있으면 에이전트가 엣지 케이스에서 스스로 판단할 수 있다. 예를 들어 파일 업로드에서 `ky`의 redirect 처리가 안 될 때, ADR에 "ky는 fetch 기반"이라는 맥락이 있으므로 `fetch`를 직접 사용하는 우회를 스스로 시도할 수 있다.

#### 2. 완료 조건이 기계적이다

```
# BAD — 주관적 완료 조건
"파일 다운로드 기능이 잘 동작하면 완료"

# GOOD — 기계적 완료 조건
"pnpm run build 성공 + dooray post file download tc-ocr 454 <file-id> 실행 시 파일 저장됨"
```

에이전트는 "잘 동작하는지"를 판단할 수 없지만, "빌드 성공 여부"와 "명령 실행 결과"는 판단할 수 있다.

#### 3. 실패 시 복구 경로가 있다

dooray-cli에서 실제 발생한 사례:

```
문제: 파일 다운로드 API가 404를 반환
원인: /files/{id}/downloads 경로가 아니라 /files/{id}?media=raw 였음
복구: 에이전트가 다양한 URL 패턴을 시도 → 정상 경로 발견 → ADR에 기록
```

이것이 가능했던 이유는 **API 문서가 컨텍스트에 포함되어 있었기 때문**이다. 에이전트가 "이 API는 `?media=raw`와 `?media=meta`를 사용한다"는 정보를 문서에서 찾아 스스로 해결했다.

### 실행 프롬프트

하네스가 준비되면 실행은 간단하다:

```
/plan-and-build
```

또는 직접 지시:

```
"docs/ 폴더를 모두 읽고, Phase 분리에 따라 v0.1.0부터 순차적으로 구현해줘.
각 Phase마다 빌드 검증하고, 실패하면 디버깅해서 통과시켜."
```

## 실제 사례: dooray-cli 타임라인

### v0.1.0 — Core CRUD (첫 대화)

```
[대화 1] 프로젝트 정의 + 기술 스택 논의
         → prd.md, adr.md, code-architecture.md 생성

[대화 2] 범위 확정 + 데이터 스키마 논의
         → data-schema.md, flow.md 생성

[대화 3] /plan-and-build 실행
         → 에이전트가 src/ 전체 구현
         → config → cache → resolver → api/client → commands 순서로 구현
         → 빌드 성공 + CLI 동작 확인
         → npm publish v0.1.0
```

### v0.2.0 — Mail (두 번째 대화)

```
[대화 4] "메일 연동도 하고 싶은데"
         → IMAP/SMTP 기술 논의
         → Dooray IMAP 서버 한계 발견 (SINCE 미지원, SORT 미지원)
         → ADR-012, ADR-013 기록

[대화 5] 에이전트가 구현 + 디버깅
         → imapflow UID 파라미터 이슈 자체 해결
         → tsup external 설정 이슈 자체 해결
         → npm publish v0.2.0
```

### v0.3.0 — File Attachment (세 번째 대화)

```
[대화 6] "파일 첨부 기능을 추가하고 싶어" + API 문서 제공
         → 307 리다이렉트 패턴 논의
         → ADR-015 기록

[대화 7] 에이전트가 구현
         → 다운로드 404 오류 → ?media=raw 경로 자체 발견
         → 업로드 ky redirect 이슈 → fetch 직접 사용으로 우회
         → 실제 API 테스트 전부 통과
         → npm publish v0.3.0
```

### 타임라인에서 보이는 패턴

1. **대화가 진행될수록 사람의 개입이 줄어든다** — v0.1.0은 논의 비중이 컸지만, v0.3.0은 API 문서만 전달하면 에이전트가 자체 해결
2. **문서가 누적되면서 에이전트의 판단력이 올라간다** — ADR, 스키마, 아키텍처 문서가 쌓일수록 에이전트가 맥락을 더 잘 이해
3. **디버깅도 자율적으로 수행** — 에러 → 원인 분석 → URL 패턴 시도 → 해결 → ADR 기록까지 자동

## 핵심 원칙 요약

### 1. 문서가 코드보다 먼저

코드를 작성하기 전에 PRD, ADR, 아키텍처 문서를 먼저 만든다. 에이전트가 참조할 수 있는 컨텍스트가 없으면 자율 실행은 불가능하다.

### 2. "왜"를 기록한다

모든 기술적 결정은 ADR로 기록한다. 결정뿐 아니라 **이유와 재검토 시점**을 함께 기록해야 에이전트가 엣지 케이스에서 판단할 수 있다.

### 3. Phase는 독립 배포 단위로

각 Phase가 끝나면 독립적으로 동작하는 제품이어야 한다. 이래야 중간에 실패해도 이전 Phase의 결과물은 보존된다.

### 4. 완료 조건은 기계적으로

"잘 동작하면 완료"가 아니라 "빌드 성공 + 이 명령 실행 시 이 결과"처럼 에이전트가 자동으로 판단할 수 있는 조건을 정의한다.

### 5. 놓친 점은 반드시 물어본다

에이전트는 효율을 추구하느라 일부를 건너뛸 수 있다. "전체 다 확인했어?"라고 명시적으로 물어야 빠진 부분을 발견할 수 있다.

### 6. 논의 → 결정 → 기록 루프

기술적 논의가 끝나면 반드시 결정을 내리고, 결정을 문서에 반영한다. 문서에 없는 결정은 다음 대화에서 잊혀진다.
